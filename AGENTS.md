# AGENTS Guide

## Scope

This repo is intentionally small. The source of truth inside the repo is `deploy-wg-easy.sh`.

Runtime state lives outside the repo under `/opt/wg-easy`.

## Important Paths

- `/root/code/wg_mk/deploy-wg-easy.sh`: deployment logic
- `/root/code/wg_mk/configure-macos-wireguard-client.sh`: macOS client-side helper
- `/root/code/wg_mk/README.md`: operator documentation
- `/opt/wg-easy/.env`: live stack settings
- `/opt/wg-easy/docker-compose.yml`: live compose file generated by the script
- `/opt/wg-easy/data/wg-easy.db`: persisted `wg-easy` application state
- `/opt/wg-easy/data/wg0.conf`: generated WireGuard server config
- `/opt/wg-easy/deployment-info.txt`: live access summary and generated admin credentials

## Repo Conventions

- Keep repo contents minimal and operational.
- Do not commit live credentials, generated `.env` files, or `/opt/wg-easy` state into the repo.
- When changing deploy behavior, update `README.md` in the same edit.
- Prefer simple Bash over introducing extra tooling.

## Deployment Behavior

- The script targets Ubuntu with `systemd`.
- It installs Docker from Docker's official apt repository.
- It clears inherited proxy environment variables unless `WG_EASY_PROXY_MODE=always`.
- In `auto` mode, it uses the local proxy only on failure paths.
- `docker compose up -d` is wrapped in a timeout. If it fails or stalls past `WG_EASY_PULL_TIMEOUT_SECONDS`, the script configures Docker and containerd to use `WG_EASY_PROXY_URL` and retries.

## Non-Obvious State

After first boot, `wg-easy` persists the advertised host in SQLite:

- DB: `/opt/wg-easy/data/wg-easy.db`
- Table: `user_configs_table`
- Column: `host`

If the public IP or DNS changes later, editing `/opt/wg-easy/.env` alone is insufficient. Update the DB value too, then recreate the container.

If clients already exist, their endpoints may also need updating:

- Table: `clients_table`
- Column: `server_endpoint`

## Safe Change Pattern

When modifying deployment behavior:

1. Update `deploy-wg-easy.sh`.
2. Run `bash -n deploy-wg-easy.sh`.
3. If the change affects usage or operations, update `README.md`.
4. If the change affects repo-specific agent guidance, update `AGENTS.md`.

## Useful Commands

Validate script syntax:

```bash
bash -n /root/code/wg_mk/deploy-wg-easy.sh
```

Check the running container:

```bash
docker ps
docker inspect --format '{{.State.Status}} {{if .State.Health}}{{.State.Health.Status}}{{end}}' wg-easy
```

Inspect live stack config:

```bash
sed -n '1,200p' /opt/wg-easy/.env
sed -n '1,220p' /opt/wg-easy/docker-compose.yml
```

Read current deployment details:

```bash
sed -n '1,200p' /opt/wg-easy/deployment-info.txt
```

Inspect persisted host value:

```bash
python3 - <<'PY'
import sqlite3
conn = sqlite3.connect('/opt/wg-easy/data/wg-easy.db')
cur = conn.cursor()
cur.execute("SELECT id, host, port FROM user_configs_table")
print(cur.fetchall())
PY
```

## Editing Guidance

- Use `apply_patch` for file edits.
- Avoid destructive git commands.
- Assume the live `/opt/wg-easy` state may matter more than the repo copy during incident work.
- If changing the default ports or host-handling logic, verify both generated files and persisted DB behavior.
